# Composer

Мультиинструментальный Telegram ИИ-агент для бота **@Composer_ubot**. Мозг — модель через OpenRouter (захардкожена в коде как `openrouter/free`): планирование, петля агента, отчёты, память. Название модели скрыто и не раскрывается ботом. Оркестрация — собственный **ReAct-цикл** (Reason → Act) без LangGraph: на каждом шаге модель возвращает строгий JSON `{thought, action, message}`, бот показывает `message` пользователю в Telegram и затем выполняет действие (модель-агностичный JSON-протокол вызова инструментов). Распознавание изображений отключено (Gemini убран).

Бот умеет: ставить точные напоминания, вести персональный memory-файл на каждого пользователя (имя, возраст, язык, стиль общения и ключевые факты — читается на каждый ответ и пополняется автоматически), искать и скачивать музыку и видео **с YouTube собственной системой на `yt-dlp`** (извлечение звука по просьбе, выбор качества), проводить обычный и глубокий research (поиск без ключей, DuckDuckGo), генерировать PDF-отчёты с кириллицей, планировать и выполнять простые многошаговые задачи, вести базу знаний.

Два режима запуска: продакшен на **Cloudflare Workers** (вебхук + Durable Objects) и временный **polling** (обычный процесс `node`, без вебхука и без публичного URL).

## Как это устроено

На Cloudflare Workers нет ни постоянного процесса, ни файловой системы, поэтому хранить «memory-файл на пользователя» и «напомнить через 20 секунд» обычный запрос-ответ не может. Решение — Durable Objects: на каждый чат поднимается один объект-актор (`ChatAgent`), у которого есть собственное durable-хранилище (там и лежит memory-файл пользователя) и точный будильник `alarm()` для напоминаний. Все апдейты одного чата маршрутизируются в один и тот же актор, поэтому состояние согласованно. Воркер лишь проверяет подпись вебхука и передаёт апдейт нужному актору. Напоминания точны до секунды и переживают засыпание изолята; для memory не нужно вручную создавать KV — хранилище Durable Object провижится миграцией при деплое (на free-плане через SQLite-бэкенд).

Загрузка медиа — собственная система на `yt-dlp` (+ `ffmpeg` для извлечения звука и склейки видео/аудио). yt-dlp — нативная программа, поэтому ей нужен настоящий процесс с файловой системой: это работает в Node-режиме (polling), но **не** в изоляте V8 на Cloudflare Workers. На Workers инструменты скачивания возвращают понятную ошибку, а `child_process` импортируется динамически, чтобы не ломать сборку воркера. Вызов инструментов сделан через строгий JSON-протокол, который парсится на нашей стороне и работает с любой моделью.

### ReAct без LangGraph

Старый state-граф в духе LangGraph убран. Теперь это простой ReAct-цикл (`src/agent/react.js`). На каждом шаге модель возвращает один JSON-объект:

```json
{ "thought": "что собираюсь сделать и почему (не показывается)",
  "action": "<имя инструмента> | respond | finish",
  "message": "человекочитаемое сообщение пользователю",
  "args": { } }
```

`message` всегда отправляется в Telegram: для шага-инструмента это короткая заметка о прогрессе ПЕРЕД выполнением («Проверяю погоду…», «Скачиваю видео…»), для `respond`/`finish` — финальный ответ. После инструмента модель получает `OBSERVATION` и решает следующий шаг. Состояние прогона (история + текущий шаг) — обычные сериализуемые данные, поэтому его можно хранить в KV и гонять по одному циклу на запрос воркера (см. ниже), либо крутить весь цикл в процессе (Durable Object и polling) через `runReact`.

### Workers + KV + Queue (опционально)

Есть необязательный режим «один ReAct-цикл = один запрос воркера»: вебхук кладёт состояние прогона в KV и ставит задачу в очередь, а consumer очереди продвигает прогон на один шаг, сохраняет состояние обратно в KV и снова ставит задачу — пока не дойдёт до `respond`. Так ни один запрос не упирается в лимит CPU. Это требует **платного** плана Workers (Queues), а в этом режиме напоминания недоступны (они живут на alarm-ах Durable Object). Включается биндингами `COMPOSER_KV` и `AGENT_QUEUE` в `wrangler.toml`; пока они не заданы, по умолчанию работает бесплатный путь через Durable Object с тем же ReAct-ядром.

## Модели: OpenRouter

Весь текст идёт через OpenRouter, модель захардкожена в `src/llm/openrouter.js` как `openrouter/free` (переменная `OPENROUTER_MODEL` игнорируется): планирование, основная петля агента, отчёты, извлечение памяти. На случай «думающих» моделей (deepseek-r1, qwen и т.п.) блок `<think>…</think>` вырезается из вывода, чтобы не ломать JSON-протокол. Распознавание изображений отключено — Gemini убран; на фото без подписи бот отвечает, что работает только с текстом. GROQ оставлен в коде, но отключён. Сменить провайдера или модель — в `src/llm/models.js` (`providerForTask`) и `src/llm/openrouter.js`; команда `/diag` показывает, доступна ли модель прямо сейчас.

## Память на пользователя

У каждого чата свой memory-файл в хранилище его Durable Object: профиль (имя, возраст, язык, стиль общения) плюс ключевые факты. Он подмешивается в системный промпт на каждый ответ, так что бот обращается по имени, отвечает на языке пользователя и держит его стиль. После каждого ответа быстрая модель тихо извлекает из сообщения новые durable-данные и дописывает их в профиль — строго то, что пользователь сказал о себе явно. Дополнительно агент может сохранять профиль и факты инструментами `update_profile` и `remember`. Команда `/memory` показывает, что бот помнит, `/forget` стирает.

## Напоминания

Естественным языком: «напомни через 20 секунд…», «напомни завтра в 9 проверить почту». Агент вызывает `set_reminder` с задержкой (секунды, минуты, часы) или абсолютным ISO-временем; объект-актор сохраняет напоминание и ставит будильник на ближайшее. Работает как будильник: когда время приходит, бот присылает сообщение с двумя инлайн-кнопками — «Остановить напоминание» и «Продлить» — и продолжает напоминать каждые 3 минуты, пока пользователь не нажмёт «Остановить». «Продлить» откладывает следующий звонок на 3 минуты. Чтобы не копились старые кнопки, у предыдущего сообщения кнопки снимаются перед новым звонком. Команда `/reminders` показывает активные, агент умеет их отменять (`cancel_reminder`). В polling-режиме напоминания работают на таймерах процесса (теряются при перезапуске).

## Скачивание и поиск

Скачивание — собственная система на `yt-dlp` (инструменты `download_media` и `download_music`), без сторонних сервисов. Поддерживается YouTube (и другие сайты, которые умеет yt-dlp). Управление через естественный язык:

- «скачай это видео» → видео в дефолтном качестве (до `YTDLP_DEFAULT_HEIGHT`, по умолчанию 1080p);
- «скачай в 720p» / «720» → видео с ограничением высоты;
- «только звук» / «вытащи аудио» → mp3 (нужен `ffmpeg`);
- «без звука» → видео без звуковой дорожки;
- «найди и скачай <артист — трек>» → `yt-dlp` сам ищет на YouTube (`ytsearch`) и достаёт аудио — без внешнего поиска.

Качество запрашивается только если его указал пользователь; иначе берётся разумный дефолт. Файлы крупнее лимита Telegram (~50 МБ) не отправляются — бот предложит аудио или меньшее качество.

Веб-поиск (`web_search`, `deep_research`) использует бесключевой HTML-эндпоинт DuckDuckGo; синтез отчёта делает та же модель. Никаких ключей Cobalt или Tavily не требуется.

```

  src/
    index.js              точка входа Worker: вебхук -> Durable Object (или очередь) + consumer очереди
    config.js             чтение и валидация переменных окружения
    llm/
      llm.js              фасад: весь текст -> OpenRouter
      openrouter.js       OpenRouter — мозг (модель openrouter/free, в коде)
      groq.js             GROQ — отключён, оставлен на будущее
      models.js           карта задача -> провайдер (единственное место правок)
    runtime/
      chatAgent.js        Durable Object: memory-файл чата + будильник напоминаний
      reminders.js        планировщик напоминаний (DO-alarms и таймеры для polling)
      polling.js          long-polling рантайм (режим без вебхуков)
      queue.js            опциональный рантайм Workers+KV+Queue (один цикл на запрос)
    agent/
      react.js            ReAct-цикл (Reason → Act): шаг, петля, парсер JSON-протокола
      run.js              сборка контекста, запуск ReAct-цикла, история, чекпойнт в KV
      systemPrompt.js     системный промпт (профиль, напоминания, секретность, протокол)
      tools.js            реестр инструментов агента
    tools/
      ytdlp.js            своя система скачивания на yt-dlp (видео/аудио, качество, поиск)
      music.js            музыка: поиск трека и извлечение аудио через yt-dlp
      research.js         web_search и deep_research (DuckDuckGo, без ключей)
      pdf.js              генерация PDF-отчётов с кириллицей
    memory/
      store.js            память/знания/планы/история/состояние прогона; KV, in-memory или DO
      extract.js          автоизвлечение профиля и фактов на быстрой модели
    telegram/             client (в т.ч. загрузка байтов), router (control + агент), format
    utils/                http и логирование без секретов
  scripts/                start-polling, env-to-devvars, sync-secrets, set-webhook
  wrangler.toml           Worker, биндинг Durable Object, миграция, опц. KV+Queue
  .env.example            шаблон единственного файла секретов
```

## Секреты: единый .env

Все секреты живут в одном файле `.env` — без `.env.local` и прочей путаницы. Обязательны `TELEGRAM_BOT_TOKEN` (от @BotFather) и `OPENROUTER_API` (openrouter.ai); `OPENROUTER_MODEL` не нужен (модель в коде); для webhook-режима ещё `TELEGRAM_WEBHOOK_SECRET` (любая случайная строка, лучше hex). Для polling его читает `npm start`; для Cloudflare `npm run dev` генерирует из него `.dev.vars`, а `npm run secrets` загружает ключи в хранилище секретов Cloudflare.

## Запуск на Cloudflare Workers

Установить зависимости `npm install`. Заполнить `.env`. Авторизоваться `npx wrangler login`. Загрузить секреты `npm run secrets`. Задеплоить `npm run deploy` — миграция сама создаст Durable Object, отдельно создавать KV не нужно. Зарегистрировать вебхук, открыв `https://<worker>/set-webhook?secret=<TELEGRAM_WEBHOOK_SECRET>`. Если деплой идёт через привязанный к Cloudflare репозиторий, секреты задаются в панели проекта (Settings → Variables and Secrets) как Secret, а не как Text — иначе их сотрёт ближайший `wrangler deploy`.

## Временный режим: polling

Заполнить в `.env` `TELEGRAM_BOT_TOKEN` и `OPENROUTER_API`, затем `npm install` и `npm start`. Бот снимет вебхук и начнёт long-polling; память и напоминания живут в памяти процесса и не переживают перезапуск. Это и есть рантайм, где работает скачивание: установи `yt-dlp` (`pip install yt-dlp`) и `ffmpeg` на хосте. На датацентровых IP YouTube иногда требует «подтвердить, что вы не робот» — тогда задай `YTDLP_COOKIES` (путь к `cookies.txt`) или `YTDLP_COOKIES_FROM_BROWSER`.

## Команды бота

`/start`, `/help`, `/memory` — что бот помнит, `/reminders` — активные напоминания, `/forget` — стереть память, `/reset` — очистить историю. Всё остальное обрабатывает агент.

## Ограничения и развитие

Скачивание работает в Node-режиме (`yt-dlp` + `ffmpeg`), не в изоляте Workers. Файлы уходят в Telegram загрузкой (multipart) с лимитом ~50 МБ (`YTDLP_MAX_FILESIZE_MB`); для крупного видео бот предложит аудио или качество пониже. Скачивание из Instagram/Pinterest пока отложено. Память и напоминания durable только на Workers (через Durable Objects); в polling-режиме — на время жизни процесса. Для периодических фоновых задач можно добавить Cron Triggers; для тяжёлой автономной работы в духе Manus — вызывать песочницу E2B по HTTPS, оставаясь на воркере.
