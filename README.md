# Composer

Мультиинструментальный Telegram ИИ-агент для бота **@Composer_ubot**. Текстовый мозг — GROQ с моделью `qwen/qwen3-32b` (планирование, петля агента, отчёты, память); распознавание изображений — Gemini, потому что у GROQ нет зрения. Название моделей скрыто и не раскрывается ботом. Оркестрация — собственный state-граф в духе LangGraph с модель-агностичным JSON-протоколом вызова инструментов.

Бот умеет: ставить точные напоминания, вести персональный memory-файл на каждого пользователя (имя, возраст, язык, стиль общения и ключевые факты — читается на каждый ответ и пополняется автоматически), искать и скачивать музыку и видео из YouTube, Instagram, Pinterest и TikTok, проводить обычный и глубокий research, генерировать PDF-отчёты с кириллицей, планировать и выполнять простые многошаговые задачи, вести базу знаний.

Два режима запуска: продакшен на **Cloudflare Workers** (вебхук + Durable Objects) и временный **polling** (обычный процесс `node`, без вебхука и без публичного URL).

## Как это устроено

На Cloudflare Workers нет ни постоянного процесса, ни файловой системы, поэтому хранить «memory-файл на пользователя» и «напомнить через 20 секунд» обычный запрос-ответ не может. Решение — Durable Objects: на каждый чат поднимается один объект-актор (`ChatAgent`), у которого есть собственное durable-хранилище (там и лежит memory-файл пользователя) и точный будильник `alarm()` для напоминаний. Все апдейты одного чата маршрутизируются в один и тот же актор, поэтому состояние согласованно. Воркер лишь проверяет подпись вебхука и передаёт апдейт нужному актору. Напоминания точны до секунды и переживают засыпание изолята; для memory не нужно вручную создавать KV — хранилище Durable Object провижится миграцией при деплое (на free-плане через SQLite-бэкенд).

Загрузка медиа делегируется по HTTPS сервису [Cobalt](https://github.com/imputnet/cobalt), потому что нативный `yt-dlp` в изоляте V8 запустить нельзя. Вызов инструментов сделан через строгий JSON-протокол, который парсится на нашей стороне и работает с любой моделью.

## Модели: GROQ qwen3-32b + Gemini (зрение)

Весь текст идёт через GROQ, модель `qwen/qwen3-32b` захардкожена в `src/llm/groq.js` (переменная `GROQ_MODEL` игнорируется): планирование, основная петля агента, отчёты, извлечение памяти. qwen3 — «думающая» модель, поэтому размышления отключены (`reasoning_effort: none`), а на всякий случай блок `<think>…</think>` ещё и вырезается из вывода, чтобы не ломать JSON-протокол. Распознавание изображений — Gemini (`GEMINI_API`), потому что у GROQ нет зрения: при получении фото бот скачивает картинку, отдаёт её Gemini, а результат уходит в ответ. Без ключа Gemini текст работает, но фото бот не распознаёт. Minimax/OpenRouter отключён. Маршрутизация — в `src/llm/models.js`, команда `/diag` показывает, какие модели доступны прямо сейчас.

## Память на пользователя

У каждого чата свой memory-файл в хранилище его Durable Object: профиль (имя, возраст, язык, стиль общения) плюс ключевые факты. Он подмешивается в системный промпт на каждый ответ, так что бот обращается по имени, отвечает на языке пользователя и держит его стиль. После каждого ответа быстрая модель тихо извлекает из сообщения новые durable-данные и дописывает их в профиль — строго то, что пользователь сказал о себе явно. Дополнительно агент может сохранять профиль и факты инструментами `update_profile` и `remember`. Команда `/memory` показывает, что бот помнит, `/forget` стирает.

## Напоминания

Естественным языком: «напомни через 20 секунд…», «напомни завтра в 9 проверить почту». Агент вызывает `set_reminder` с задержкой (секунды, минуты, часы) или абсолютным ISO-временем; объект-актор сохраняет напоминание и ставит будильник на ближайшее. Работает как будильник: когда время приходит, бот присылает сообщение с двумя инлайн-кнопками — «Остановить напоминание» и «Продлить» — и продолжает напоминать каждые 3 минуты, пока пользователь не нажмёт «Остановить». «Продлить» откладывает следующий звонок на 3 минуты. Чтобы не копились старые кнопки, у предыдущего сообщения кнопки снимаются перед новым звонком. Команда `/reminders` показывает активные, агент умеет их отменять (`cancel_reminder`). В polling-режиме напоминания работают на таймерах процесса (теряются при перезапуске).

## Структура проекта

```
composer/
  src/
    index.js              точка входа Worker: вебхук -> Durable Object актор
    config.js             чтение и валидация переменных окружения
    llm/
      llm.js              фасад: текст -> GROQ, зрение -> Gemini
      groq.js             GROQ qwen/qwen3-32b — текстовый мозг (модель в коде)
      gemini.js           Gemini: распознавание изображений (vision)
      openrouter.js       Minimax через OpenRouter (отключён, оставлен на будущее)
      models.js           карта задача -> провайдер (единственное место правок)
    runtime/
      chatAgent.js        Durable Object: memory-файл чата + будильник напоминаний
      reminders.js        планировщик напоминаний (DO-alarms и таймеры для polling)
      polling.js          long-polling рантайм (режим без вебхуков)
    agent/
      graph.js            движок state-графа + петля агента (JSON tool-calling)
      run.js              сборка контекста, запуск графа, история
      systemPrompt.js     системный промпт (профиль, напоминания, секретность)
      tools.js            реестр инструментов агента
    tools/                media, music, research, pdf
    memory/
      store.js            память/знания/планы/история; KV, in-memory или DO-хранилище
      extract.js          автоизвлечение профиля и фактов на быстрой модели
    telegram/             client, router (команды + диспетчеризация), format
    utils/                http и логирование без секретов
  scripts/                start-polling, env-to-devvars, sync-secrets, set-webhook
  wrangler.toml           Worker, биндинг Durable Object и миграция
  .env.example            шаблон единственного файла секретов
```

## Секреты: единый .env

Все секреты живут в одном файле `.env` — без `.env.local` и прочей путаницы. Обязательны `TELEGRAM_BOT_TOKEN` (от @BotFather) и `GROQ_API` (console.groq.com); `GEMINI_API` опционален и нужен только для распознавания фото; для webhook-режима ещё `TELEGRAM_WEBHOOK_SECRET` (любая случайная строка, лучше hex). Для polling его читает `npm start`; для Cloudflare `npm run dev` генерирует из него `.dev.vars`, а `npm run secrets` загружает ключи в хранилище секретов Cloudflare.

## Запуск на Cloudflare Workers

Установить зависимости `npm install`. Заполнить `.env`. Авторизоваться `npx wrangler login`. Загрузить секреты `npm run secrets`. Задеплоить `npm run deploy` — миграция сама создаст Durable Object, отдельно создавать KV не нужно. Зарегистрировать вебхук, открыв `https://<worker>/set-webhook?secret=<TELEGRAM_WEBHOOK_SECRET>`. Если деплой идёт через привязанный к Cloudflare репозиторий, секреты задаются в панели проекта (Settings → Variables and Secrets) как Secret, а не как Text — иначе их сотрёт ближайший `wrangler deploy`.

## Временный режим: polling

Заполнить в `.env` `TELEGRAM_BOT_TOKEN` и `GROQ_API` (по желанию `GEMINI_API` для фото), затем `npm install` и `npm start`. Бот снимет вебхук и начнёт long-polling; память и напоминания живут в памяти процесса и не переживают перезапуск.

## Команды бота

`/start`, `/help`, `/memory` — что бот помнит, `/reminders` — активные напоминания, `/forget` — стереть память, `/reset` — очистить историю. Всё остальное обрабатывает агент.

## Ограничения и развитие

Telegram принимает медиа по URL примерно до 20 МБ и по загрузке до 50 МБ — крупные файлы стоит отдавать ссылкой. Память и напоминания durable только на Workers (через Durable Objects); в polling-режиме — на время жизни процесса. Для периодических фоновых задач можно добавить Cron Triggers; для тяжёлой автономной работы в духе Manus — вызывать песочницу E2B по HTTPS, оставаясь на воркере.
